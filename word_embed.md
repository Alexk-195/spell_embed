# Compact Character-Level Word Embedding

This document describes a **compact, training-free algorithm for embedding words** such that:

* Small misspellings or missing letters produce **small changes** in the embedding
* Reordering characters or using unrelated words produces **large changes**
* The algorithm is **deterministic**, lightweight, and does **not rely on semantic models**

The method is purely **orthographic** and operates at the **character n-gram** level. It draws on ideas from **reservoir computing** (Echo State Networks) and **random feature maps** for string kernels.

---

## 1. Design Goals

We want an embedding function:

```
word → v ∈ ℝᵈ
```

with the following properties:

* Edit tolerance (insertions, deletions, substitutions, transpositions)
* Strong order sensitivity (anagrams and reversals are far apart)
* Suffix-friendly (adding "s", "ed", "ing" produces small changes)
* Fixed embedding dimension
* No training or large parameter sets

This is useful for:

* Spell checking / fuzzy matching
* Identifier or token similarity
* Lightweight NLP pipelines
* Typo-tolerant search

---

## 2. Character Encoding

Let the alphabet size be `K` (e.g. 31: 26 lowercase letters + 4 German special characters öäüß + 1 start boundary marker).

By default, each character is encoded as a **one-hot vector**:

```
a → [1, 0, 0, ..., 0]
b → [0, 1, 0, ..., 0]
...
^ → [0, ..., 0, 1]       (start boundary)
```

When **character similarities** are defined (see Section 6), characters with single-character similarity targets use **soft encoding** instead: a normalized blend of the character's own one-hot with the target's one-hot, weighted by the similarity strength. Characters without similarity entries retain pure one-hot vectors — behavior is identical to the unmodified algorithm.

---

## 3. Start Boundary Marker

Words are prefixed with an explicit start boundary before n-gram extraction:

```
"guarantee" → "^guarantee"
"cat"       → "^cat"
```

The start marker `^` is included in the alphabet (e.g. `K = 31` with German characters, or `K = 27` for plain English).

Only a **start boundary** is used (no end marker `$`). This is a deliberate design choice: an end marker would shift position when suffixes are added (e.g. `orange$` vs `oranges$`), causing all n-grams near the end to differ. Without `$`, adding a suffix only **appends new n-grams** without modifying existing ones, keeping morphological variants close together.

---

## 4. Multi-Scale Sliding n-grams

Words are split into overlapping n-grams at **multiple scales** (default: `n = 2, 3, 4`):

```
"^cat" →
  bigrams:  ^c, ca, at
  trigrams: ^ca, cat
  4-grams:  ^cat
```

N-grams are collected **position by position, all scales at each position**:

```
position 0: ^c, ^ca, ^cat
position 1: ca, cat
position 2: at
```

This interleaving ensures the recurrent state sees fine-grained (bigram) and coarse (4-gram) signals at each position before advancing. Multi-scale n-grams are important because:

* **Bigrams** capture single-character context — critical for short words
* **Trigrams** capture local character patterns
* **4-grams** capture longer structural features — stable under small edits in long words

---

## 5. n-gram Vector Construction

For an n-gram `(c₁, c₂, ..., cₙ)`, construct a vector by **concatenating** the encoding vectors:

```
x = [encode(c₁) || encode(c₂) || ... || encode(cₙ)] ∈ ℝⁿᴷ
```

Where `encode(c)` is the one-hot vector for `c`, or the soft encoding vector if character similarities are defined for `c` (see Section 6).

This representation is order-aware. Each n-gram scale has a different input dimension (`n * K`).

---

## 6. Character Similarity

An optional similarity dictionary maps characters to a list of `(target, weight)` pairs:

```
similarities = {
    'ö': [('o', 0.7), ('oe', 0.5)],
    'ä': [('a', 0.7), ('ae', 0.5)],
    'ü': [('u', 0.7), ('ue', 0.5)],
    'ß': [('ss', 0.7)],
}
```

This enables two complementary mechanisms:

### Single-character targets → soft encoding

When the target is a single character (e.g. `ö → o` with weight 0.7), the encoding vector for the source character is blended with the target's one-hot:

```
encode('ö') = normalize( onehot('ö') + 0.7 · onehot('o') )
```

Every n-gram containing `ö` automatically becomes close to the same n-gram with `o`, because the projected vectors `B · x` are similar. This requires no structural changes — it operates entirely within the character encoding layer.

### Multi-character targets → expansion blending

When the target is a string of length > 1 (e.g. `ö → oe`, `ß → ss`), the word lengths differ and n-gram alignments shift. No local encoding trick can fix this. Instead, alternate word forms are generated by replacing each occurrence of the source character with its multi-character expansion:

```
"König"  → expansion: "Koenig"  (ö → oe, weight 0.5)
"Straße" → expansion: "Strasse" (ß → ss, weight 0.7)
```

Each expansion is embedded separately and blended with the original:

```
v = (v_original + w₁ · v_expansion₁ + w₂ · v_expansion₂ + ...) / (1 + w₁ + w₂ + ...)
```

This weighted average pulls the embedding of `König` toward `Koenig` proportionally to the similarity weight.

### Zero overhead for unaffected words

Characters without similarity entries get pure one-hot encoding. Words containing no similarity-mapped characters follow the exact same code path as before — the only added cost is an O(1) dictionary lookup per character.

---

## 7. Sparse Contractive State Matrix

A state transition matrix `A` is constructed as a **sparse contractive random matrix**:

```
1. Sample A from N(0, 1)  ∈ ℝᵈˣᵈ
2. Apply sparsity mask: A = A ⊙ mask   (mask has ~15% nonzero entries)
3. Rescale: A = α · A / ‖A‖₂           (spectral norm normalization)
```

Where `α < 1` is the target spectral radius (default: `α = 0.9`).

**Why sparse?** Inspired by Echo State Network (ESN) theory, sparse connectivity creates **heterogeneous reservoir dynamics** — different dimensions of the state vector respond to different character patterns. Dense random matrices produce more homogeneous dynamics with less discriminative power.

**Why contractive?** The spectral norm constraint `‖A‖₂ < 1` guarantees that the recurrent dynamics don't blow up over long words. The state remains bounded regardless of input length.

---

## 8. Hybrid Sequential + Bag-of-Ngrams Architecture

The embedding combines two complementary components:

### Sequential pass (left → right)

Captures n-gram **order** via non-commutative recurrent dynamics:

```
v₀_seq = 0
vₖ₊₁_seq = vₖ_seq + tanh( A · vₖ_seq + Bₙ · xₖ )
```

### Bag-of-ngrams pass

Captures n-gram **composition** — the sum of all projected n-gram vectors:

```
v_bag_sum = Σₖ Bₙ · xₖ
```

Where:

* `vₖ ∈ ℝᵈ` is the state after processing the k-th n-gram
* `A ∈ ℝᵈˣᵈ` is a **sparse contractive matrix**
* `Bₙ ∈ ℝᵈˣⁿᴷ` is the **random projection matrix** for n-gram scale `n`
* `T` is the total number of n-grams processed across all scales
* `tanh` bounds the contribution of each n-gram to `[-1, +1]` per dimension

### Final combination with configurable length normalization

```
v = v_seq / T^(0.5 · λ) + v_bag_sum / T^λ
```

Where `λ` (`length_norm`) is a parameter in `[0, 1]` that controls length normalization strength:

* `λ = 1.0` (default): full normalization — `v_seq / √T + v_bag_sum / T`. Long words produce smaller per-edit distances. Edits are measured relative to word length.
* `λ = 0.0`: no normalization — `v_seq + v_bag_sum`. Per-edit distances are approximately independent of word length.
* Intermediate values interpolate smoothly between these behaviors.

**Why hybrid?** The sequential component ensures anagrams and reversals (same characters, different order) produce different embeddings. The bag component ensures words sharing most of their n-grams — like `orange` and `oranges` — remain close, since the bag sum shifts only slightly when a suffix adds a few new n-grams. This combination makes suffix additions much cheaper than mid-word edits.

---

## 9. Why the Residual Update Matters

Using a direct (non-residual) update:

```
vₖ₊₁ = tanh(A vₖ + B xₖ)
```

causes **state saturation**: after several steps, `tanh` drives all states toward `[-1, +1]` boundaries, making different words collapse to similar embeddings.

The **residual form**:

```
vₖ₊₁ = vₖ + tanh(A vₖ + B xₖ)
```

turns the embedding into a **path integral over n-grams**, ensuring:

* Each n-gram contributes a bounded increment (via `tanh`)
* The state grows with word length rather than saturating
* Missing or extra n-grams cause incremental (not catastrophic) changes
* Edit distance maps smoothly to L2 distance

## 10. Default Parameters

| Parameter      | Default | Description |
|----------------|---------|-------------|
| `d`            | 256     | Embedding dimension |
| `scales`       | (2,3,4) | N-gram sizes (bigrams, trigrams, 4-grams) |
| `alpha`        | 0.9     | Spectral radius of A matrix |
| `sparsity`     | 0.85    | Fraction of zeros in A (85% sparse) |
| `seed`         | 0       | Random seed for reproducibility |
| `alphabet`     | a-z^öäüß | 26 letters + 4 German chars + 1 start boundary marker (K=31) |
| `similarities` | None    | Character similarity dict (see Section 6) |
| `length_norm`  | 1.0     | Length normalization strength: 0.0 = none, 1.0 = full (see Section 8) |

**Key takeaway:** For **string similarity** tasks (spell checking, fuzzy matching, typo-tolerant search), our method significantly outperforms compressed FastText at **200x smaller model size** with **no training**. For non-English text, FastText's subword hashing can produce degenerate results (0.00 distances for unrelated OOV words), while our method handles arbitrary alphabets reliably.

## 13. Characteristics and Limitations

### Strengths

* No training required — fully deterministic given a random seed
* Small, fixed parameter set (one sparse d × d matrix + a few d × nK matrices)
* Strong order sensitivity via non-commutative recurrent dynamics
* Smooth edit tolerance via residual path integral
* Suffix-friendly via bag-of-ngrams component
* Configurable length normalization via `length_norm` parameter
* Character similarity support for languages with special characters (umlauts, ß, etc.)
* Handles both single-char (soft encoding) and multi-char (expansion blending) equivalences
* Zero overhead for words not containing similarity-mapped characters
* 200x smaller than compressed FastText with 3.4x better string similarity

### Limitations

* No semantic understanding (`happy` and `sad` are far apart; `cat` and `dog` are far apart)
* Character-level only (not subword or morpheme-aware), though character similarity partially bridges this for known equivalences
* Embedding quality depends on the random seed — no formal optimality guarantee
* Edit distance cannot be perfectly embedded into Euclidean space in general (Ostrovsky & Rabani, 2007)
* Trained methods (FastText, character CNNs/RNNs) will outperform on semantic benchmarks
* Position symmetry is approximate — start-of-word edits produce slightly larger distances than end-of-word edits due to forward-only sequential pass
* Expansion blending generates one variant per expandable character — words with many special characters produce multiple embeddings, adding linear cost

This is a **string similarity embedding**, not a language model.

## 14. Related Work

## 15. Summary

This method provides a compact, training-free way to embed words using:

* Multi-scale character n-grams (bigrams, trigrams, 4-grams)
* Hybrid sequential + bag-of-ngrams architecture
* Sparse contractive recurrence with random projections
* Configurable length normalization (`length_norm` parameter)
* Character similarity support via soft encoding (single-char) and expansion blending (multi-char)

It achieves **3.4x better string similarity separation** than compressed FastText for English (and **3.3x** for German with character similarities) at **200x smaller model size**, without any training data. It sits between classic string kernels and learned character RNNs, offering a practical tradeoff between **simplicity**, **robustness**, and **expressiveness** for applications where training data is unavailable or unnecessary.
